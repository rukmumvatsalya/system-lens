Wiom Design System Foundation Review

A structured audit of the Wiom Next typography system (13 semantic tokens), color system (39 semantic tokens), shadow system (5 tokens), radius system (7 tokens), and stroke system (2 width tokens) against world-class design systems — identifying what's strong, what's missing, and what to prioritize next.
Input: wiom_typography_foundation.html + index.html (color) + stroke foundation Scope: Typography + Color + Shadow + Radius + Stroke tokens Platform: Mobile (Android), Bilingual (EN + HI) Date: April 2026 Agent: System Lens v1.0
Overall Conformance Conditional Pass
Strong foundation across typography, color, shadow, radius, and stroke — scale, naming, weight logic, bilingual calibration, migration plan, color architecture, semantic naming, interaction states, shadow system, radius system, and stroke system are at or above industry standard. Typography has 4 HIGH-priority gaps (letter-spacing, token architecture, font-scaling, type-color mapping). Color has 1 CRITICAL gap (Usage Guide inconsistency with Token Spec), plus missing dark mode palette, data visualization colors, and opacity tokens. Shadow is exceptionally clean with unique strengths (single formula, decision tree, budget device guidance) but missing dark mode shadow adjustments. Radius is exceptionally thorough — 7 tokens on 4px grid, aliased to spacing primitives, with unique strengths (nested formula with token pairs, 8 anti-patterns, component mapping) that exceed every benchmark system. Stroke is the most comprehensive border foundation in the benchmark set — 2 width tokens with clear passive/interactive split, "filled = no border" rule, 11 anti-patterns, Android-specific focus ring guidance, and "shadow vs border: never both" rule are all unique strengths. Only missing dark mode border colors and border transition timing.

Detailed Findings
Typography Token Audit
Each area of the typography system reviewed against world-class benchmarks.
Type Scale
PASS
What's Here
8 clean sizes: 10 · 12 · 14 · 16 · 20 · 24 · 32 · 48. Clear jumps, no ambiguous intermediates (eliminated 11, 13, 15, 18, 22, 28). 50% reduction from 24 legacy tokens to 13. Scale calibrated for Devanagari readability on budget Android.
Strength: The scale matches industry consensus perfectly. Carbon uses 14, Polaris ~13, Primer ~12, Material Design 3 uses 15. Wiom's 13 is right in the sweet spot. The bilingual calibration and budget-device awareness are unique strengths no benchmark system offers.
Naming Convention
PASS
What's Here
type.{role}.{size} — semantic, role-based. 6 roles: display, heading, title, body, label, meta. Makes token selection self-documenting for both humans and AI tools.
Benchmark: Matches Carbon ($heading-01), Polaris (--p-font-size-heading-lg), Material Design 3 (M3/display/large). 10/10 benchmarked systems use role-based naming. Wiom's dot notation is clean and scannable.
Weight System
PASS
What's Here
4 weights (Regular 400, Medium 500, SemiBold 600, Bold 700) with clear role assignments. Weight-gap analysis ensures ≥200 gap at shared sizes, with one documented exception (16px: Bold/SemiBold/Regular for isolated button context).
Weight-Role Mapping
Bold 700 = structural headings, titles  ·  SemiBold 600 = interactive labels  ·  Medium 500 = navigational titles  ·  Regular 400 = readable body content
Benchmark: Most systems use 3-4 weights. Material Design 3 uses Regular/Medium. Carbon uses 4 (300/400/600/700). Wiom's "weight carries role at shared sizes" approach is on par with Fluent 2. The SemiBold + Medium prohibition at same size is well-reasoned for budget Android.
Line Height / Rhythm
PASS
What's Here
Every line-height is a 4px multiple: 16 · 20 · 24 · 28 · 32 · 40 · 56. Vertical rhythm stays consistent in layouts.
Benchmark: 10/10 benchmarked systems enforce 4px or 8px grid. Wiom aligns perfectly.

What's Missing

Suggestion: Document the primitive layer underneath (even if simple — font.size.48, font.weight.700) and the component bindings above (e.g., button.text → type.label.l, card.title → type.title.s). This makes the system extensible and auditable.
Letter Spacing
MISSING
What's Here
One instance of letter-spacing: -0.02em in the hero CSS. No systematic letter-spacing tokens per role.
What's Missing
Letter-spacing tokens per type scale step. At larger sizes (32px, 48px), tighter tracking improves readability. At smaller sizes (10px, 12px), slightly wider tracking helps legibility — especially critical for Devanagari script.
Benchmark: Material Design 3 defines letter-spacing for every type role (display: -0.25px, body: +0.25px). Carbon defines tracking per scale step. Apple HIG recommends tighter tracking for display and looser for body.
Suggestion: Add a letter-spacing value to each of the 13 tokens. At minimum: negative for display/heading, zero for body, slightly positive for label/meta.
Responsive Typography
MISSING
What's Here
No mention of how the type scale adapts across screen sizes, viewport widths, or user font-size preferences.
What's Missing
• No breakpoint-specific scale adjustments (e.g., display.l = 48px on large, 32px on small)
• No documentation for tablet vs phone behavior
• No fluid or responsive type strategy
Benchmark: Apple HIG has Dynamic Type with 7 accessibility sizes. Material Design 3 has density-aware typography. Fluent 2 has a responsive type ramp that adjusts per viewport.
Suggestion: Document: (1) whether these sizes are fixed or adapt, (2) how the system handles Android's font-size accessibility settings, (3) whether display tokens behave differently on tablets.
Accessibility (Typography)
GAP
What's Here
Excellent bilingual consideration: Devanagari readability at small sizes, meta.xs restriction (timestamps/badges only), 12px minimum for Hindi body text, budget device awareness. This is rare and strong.
What's Missing
• Font scaling / Dynamic Type: No documentation on how the system responds to Android accessibility font-size settings. What happens when a user sets "Large text"?
• Dark mode weight adjustment: No mention of whether weight changes on dark backgrounds (light-on-dark text appears thinner — some systems bump weight by 100).
• Contrast guidance: No formalized minimum contrast ratios per typography role.
Benchmark: Apple HIG's Dynamic Type scales automatically with user preferences (gold standard). Paste (Twilio) documents minimum contrast per type role. Carbon targets WCAG AAA for all text.
Suggestion: Document: (1) Android sp vs dp strategy — are tokens in sp (scales with preference) or dp (fixed)? (2) Dark mode typography behavior. (3) Minimum contrast ratio per token role.
Color × Typography
MISSING
What's Here
The specimen uses --n900 for primary text and --n600 / --n500 for secondary/meta text, but this is in the HTML presentation only — not formalized as part of the token system.
What's Missing
Formal mapping of color tokens to typography roles:
• Heading text color
• Body text color (primary, secondary)
• Label text color (enabled, disabled, focus states)
• Meta/tertiary text color
• Error, success, warning text colors
• Color-on-surface relationships
Benchmark: Carbon has $text-primary, $text-secondary, $text-disabled mapped to type roles. Material Design 3 has on-surface, on-surface-variant semantic color tokens explicitly for text. Polaris maps color to component contexts.
Suggestion: Define semantic text-color tokens (text.primary, text.secondary, text.disabled, text.error) and document which typography roles default to which color.
Documentation & Migration
PASS
What's Here
• 8 legacy gaps identified with industry evidence
• Before/after comparison ("3 devs, 3 results → 3 devs, 1 result")

Strength: This is above-benchmark documentation. Most design systems don't document the why behind decisions. Matches Carbon's depth (best in class). Exceeds Polaris, Primer, and Material Design 3 in migration documentation. The bilingual specimen is rare — most systems only show Latin script.
