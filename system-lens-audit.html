Detailed Findings
Typography Token Audit
Each area of the typography system reviewed against world-class benchmarks.
Type Scale
PASS
What's Here
8 clean sizes: 10 · 12 · 14 · 16 · 20 · 24 · 32 · 48. Clear jumps, no ambiguous intermediates (eliminated 11, 13, 15, 18, 22, 28). 50% reduction from 24 legacy tokens to 13. Scale calibrated for Devanagari readability on budget Android.
Strength: The scale matches industry consensus perfectly. Carbon uses 14, Polaris ~13, Primer ~12, Material Design 3 uses 15. Wiom's 13 is right in the sweet spot. The bilingual calibration and budget-device awareness are unique strengths no benchmark system offers.
Naming Convention
PASS
What's Here
type.{role}.{size} — semantic, role-based. 6 roles: display, heading, title, body, label, meta. Makes token selection self-documenting for both humans and AI tools.
Benchmark: Matches Carbon ($heading-01), Polaris (--p-font-size-heading-lg), Material Design 3 (M3/display/large). 10/10 benchmarked systems use role-based naming. Wiom's dot notation is clean and scannable.
Weight System
PASS
What's Here
4 weights (Regular 400, Medium 500, SemiBold 600, Bold 700) with clear role assignments. Weight-gap analysis ensures ≥200 gap at shared sizes, with one documented exception (16px: Bold/SemiBold/Regular for isolated button context).
Weight-Role Mapping
Bold 700 = structural headings, titles  ·  SemiBold 600 = interactive labels  ·  Medium 500 = navigational titles  ·  Regular 400 = readable body content
Benchmark: Most systems use 3-4 weights. Material Design 3 uses Regular/Medium. Carbon uses 4 (300/400/600/700). Wiom's "weight carries role at shared sizes" approach is on par with Fluent 2. The SemiBold + Medium prohibition at same size is well-reasoned for budget Android.
Line Height / Rhythm
PASS
What's Here
Every line-height is a 4px multiple: 16 · 20 · 24 · 28 · 32 · 40 · 56. Vertical rhythm stays consistent in layouts.
Benchmark: 10/10 benchmarked systems enforce 4px or 8px grid. Wiom aligns perfectly.
Token Architecture
GAP
What's Here
13 flat semantic tokens with usage descriptions. Clear roles and values documented.
What's Missing
No documented layered architecture (primitive → alias → component). World-class systems work in 3 layers:

Benchmark: Carbon documents all 3 layers explicitly. Material Design 3 maps tokens through sys.typescale.display-large → component usage. Polaris has component-level token bindings in their docs.
Suggestion: Document the primitive layer underneath (even if simple — font.size.48, font.weight.700) and the component bindings above (e.g., button.text → type.label.l, card.title → type.title.s). This makes the system extensible and auditable.
Letter Spacing
MISSING
What's Here
One instance of letter-spacing: -0.02em in the hero CSS. No systematic letter-spacing tokens per role.
What's Missing
Letter-spacing tokens per type scale step. At larger sizes (32px, 48px), tighter tracking improves readability. At smaller sizes (10px, 12px), slightly wider tracking helps legibility — especially critical for Devanagari script.
Benchmark: Material Design 3 defines letter-spacing for every type role (display: -0.25px, body: +0.25px). Carbon defines tracking per scale step. Apple HIG recommends tighter tracking for display and looser for body.
Suggestion: Add a letter-spacing value to each of the 13 tokens. At minimum: negative for display/heading, zero for body, slightly positive for label/meta.
Responsive Typography
MISSING
What's Here
No mention of how the type scale adapts across screen sizes, viewport widths, or user font-size preferences.
What's Missing
• No breakpoint-specific scale adjustments (e.g., display.l = 48px on large, 32px on small)
• No documentation for tablet vs phone behavior
• No fluid or responsive type strategy
Benchmark: Apple HIG has Dynamic Type with 7 accessibility sizes. Material Design 3 has density-aware typography. Fluent 2 has a responsive type ramp that adjusts per viewport.
Suggestion: Document: (1) whether these sizes are fixed or adapt, (2) how the system handles Android's font-size accessibility settings, (3) whether display tokens behave differently on tablets.
Accessibility (Typography)
GAP
What's Here
Excellent bilingual consideration: Devanagari readability at small sizes, meta.xs restriction (timestamps/badges only), 12px minimum for Hindi body text, budget device awareness. This is rare and strong.
What's Missing
• Font scaling / Dynamic Type: No documentation on how the system responds to Android accessibility font-size settings. What happens when a user sets "Large text"?
• Dark mode weight adjustment: No mention of whether weight changes on dark backgrounds (light-on-dark text appears thinner — some systems bump weight by 100).
• Contrast guidance: No formalized minimum contrast ratios per typography role.
Benchmark: Apple HIG's Dynamic Type scales automatically with user preferences (gold standard). Paste (Twilio) documents minimum contrast per type role. Carbon targets WCAG AAA for all text.
Suggestion: Document: (1) Android sp vs dp strategy — are tokens in sp (scales with preference) or dp (fixed)? (2) Dark mode typography behavior. (3) Minimum contrast ratio per token role.
Color × Typography
MISSING
What's Here
The specimen uses --n900 for primary text and --n600 / --n500 for secondary/meta text, but this is in the HTML presentation only — not formalized as part of the token system.
What's Missing
Formal mapping of color tokens to typography roles:
• Heading text color
• Body text color (primary, secondary)
• Label text color (enabled, disabled, focus states)
• Meta/tertiary text color
• Error, success, warning text colors
• Color-on-surface relationships
Benchmark: Carbon has $text-primary, $text-secondary, $text-disabled mapped to type roles. Material Design 3 has on-surface, on-surface-variant semantic color tokens explicitly for text. Polaris maps color to component contexts.
Suggestion: Define semantic text-color tokens (text.primary, text.secondary, text.disabled, text.error) and document which typography roles default to which color.
Documentation & Migration
PASS
What's Here
• 8 legacy gaps identified with industry evidence
• Before/after comparison ("3 devs, 3 results → 3 devs, 1 result")
• Decision rationale for every choice (scale, weight, naming)
• Migration mapping (keep 13 / merge 5 / legacy 1 / remove 6)
• Screen-by-screen migration examples
• 4-tier rollout plan (Keep / Merge / Legacy / Remove)
• Bilingual type specimen (all 13 tokens in EN + HI)
• AI workflow consistency considerations
Strength: This is above-benchmark documentation. Most design systems don't document the why behind decisions. Matches Carbon's depth (best in class). Exceeds Polaris, Primer, and Material Design 3 in migration documentation. The bilingual specimen is rare — most systems only show Latin script.

Color Token Audit
Each area of the color system reviewed against world-class benchmarks.
Color Palette & Scale
PASS
What's Here
33 primitives + 39 semantic tokens across 8 groups (Brand 7, Text 7, Surface 5, Border 4, Positive 4, Negative 6, Warning 3, Info 4, Overlay 1). Consistent primary/soft/subtle/tint scale across all semantic families.
Strength: The consistent scale pattern is excellent. Every semantic family follows the same graduation (primary → soft → subtle → tint), making the system predictable. Matches Carbon and Polaris patterns exactly. The brand family has a coherent hue (328°) from secondary (#4A1535) through primary (#D9008D) to tint (#FFE5F6).
Color Naming
PASS
What's Here
color.{family}.{role} — semantic, role-based. Families: brand, text, surface, border, positive, negative, warning, info, overlay.
Benchmark: Matches Carbon ($color-text-primary), Polaris (--p-color-text), Material Design 3 (sys.color.on-surface). 10/10 systems use role-based naming.
Color Architecture
PASS
What's Here
2-tier system (primitives for theming, semantic tokens for UI). Tier 3 (component tokens) planned. Primitives are NOT directly used in UI — developers only interact with semantic tokens.
Strength: This is the correct architecture. Matches Carbon, Material Design 3, Polaris, and Atlassian exactly. The planned Tier 3 for component tokens is forward-thinking. The "minimize the decision surface, not the total architecture" principle is sound.
Benchmark: Carbon, MD3, Polaris, Atlassian all use this same layered approach.
Interaction States
PASS
What's Here
Brand hover (#C00080), brand pressed (#A30070), negative hover (#BF1D2A), negative pressed (#A31824). Secondary/tertiary button states use brand tint/subtle. Focus ring unified to border.brand (#FFB2E4), destructive focus uses negative.soft (#FFB3B9). Disabled: text.disabled + surface.muted + 40% opacity on fills.
Strength: Comprehensive coverage of all interactive states with clear decision tree.
Benchmark: Matches Polaris and Atlassian approach — focus identity at brand level, not per-semantic-family.
Contrast Pairings
PASS
What's Here
10 approved safe pairings with WCAG ratios documented. 4 "avoid" pairings listed. Mobile-specific guidance for brand primary on white (4.84:1 — passes AA, fails AAA). Recommendation to use brand.primary-hover (#C00080, ~5.5:1) for small brand-colored text on budget devices.
Strength: The mobile-specific contrast guidance is rare and valuable for Wiom's audience.
Benchmark: Carbon targets WCAG AAA. Paste (Twilio) documents per-component contrast. Wiom's AA documentation is solid; AAA would be aspirational for future.
Decision Trees
PASS
What's Here
Text, surface, border, and interaction state selection documented with clear decision trees.
Strength: Decision trees reduce guesswork for developers. This is a mature pattern seen in Carbon and Polaris.
Document Consistency
GAP
What's Here
The Token Spec tab shows the post-decision 39-token system. The Usage Guide tab still shows pre-decision tokens.
What's Wrong
• Usage Guide shows text.tertiary (#665E75) — removed in Decision 004
• Usage Guide shows 5 text tokens — Token Spec has 7 (includes link, link-hover, on-color)
• Usage Guide shows Warning as orange (#FF8000) + separate Caution (yellow) — merged in Decision 006
• Usage Guide shows brand.secondary as #443152 — changed to #4A1535 in Decision 008
• Usage Guide shows text.secondary as #352D42 — changed to #5C5570 in Decision 004
• Quick reference table lists 38 tokens + 1 gradient effect vs header's "39 semantic tokens"
This is the most critical finding. A developer reading the Usage Guide will implement wrong tokens. The Usage Guide must be updated to match the Token Spec post-decisions.
Suggestion: Rebuild Usage Guide from Token Spec as source of truth. Add a version stamp to each tab so readers know which is current.
Dark Mode
MISSING
What's Here
Architecture supports mode-swap by reassigning primitives. Gap 8 in legacy audit marked "Ready."
What's Missing
No dark palette defined. No dark mode primitives proposed. No dark-on-light contrast pairings.
Benchmark: Carbon ships light + dark from day 1. Material Design 3 generates dark theme automatically via tonal palette. Fluent 2 has 3 themes. Polaris ships light + dark.
Suggestion: Define dark mode primitives and semantic token overrides. At minimum, document the primitive swap table (light → dark).
Data Visualization Colors
MISSING
What's Here
No chart, graph, or data visualization palette.
What's Missing
Categorical colors for charts/graphs, sequential/diverging palettes, colorblind-safe data colors.
Benchmark: Carbon has a full data visualization palette (14 categorical colors). Polaris has chart colors. Stripe, Groww, Revolut all have dedicated data viz palettes.
Suggestion: Add 6-8 categorical data colors that are distinguishable, colorblind-safe, and consistent with the brand palette.
Positive/Warning/Info Interaction States
GAP
What's Here
Brand and Negative both have hover/pressed tokens. Positive, Warning, and Info do NOT.
What's Missing
positive.primary-hover, positive.primary-pressed (for "confirm" style buttons). Warning and info have no interaction variants.
Benchmark: Most systems don't have hover/pressed for semantic colors beyond brand + destructive. This is a minor gap.
Suggestion: Add positive hover/pressed only if positive-colored buttons are planned. Warning/info interaction states are low priority.
Opacity Tokens
MISSING
What's Here
Disabled states use "40% opacity on base color" and overlay uses rgba with 0.50. No formalized opacity tokens.
What's Missing
Opacity scale tokens (e.g., opacity.disabled = 0.4, opacity.overlay = 0.5, opacity.hover = 0.08).
Benchmark: Fluent 2 defines opacity tokens. Carbon has opacity variables. Most systems formalize this.
Suggestion: Define at minimum: opacity.disabled (0.4), opacity.overlay (0.5), opacity.hover-tint (0.08).
Documentation & Legacy Audit
PASS
What's Here
12 legacy gaps identified and resolved. 9 design decisions logged with rationale and industry benchmarks. Decision trees for text, surface, border, and interaction state selection. Do/Don't guidelines. Implementation files documented.
Strength: The decision log is exceptional — every decision has a "problem → industry pattern → Wiom decision" structure. Decision 006 (Warning/Caution merge) is particularly well-reasoned with 10/10 system benchmarking. Exceeds most design systems' documentation.
